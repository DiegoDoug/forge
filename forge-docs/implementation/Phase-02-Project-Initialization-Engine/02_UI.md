# Project Initialization Engine — UI

> **Purpose:** Screen-level and interaction-level UI specification for this phase.
> **Scope:** UI/UX only. Component implementation detail lives in 05_COMPONENTS.md; general UI conventions live in the root UI guidelines.
> **Ownership:** Lead Software Engineer (session-assigned).
> **Status:** Accepted.
> **Last Updated:** 2026-07-22

---

## 1. Screens / views

Single page: `/project-init`, titled "Project Init" in nav, sidebar icon `Sparkles` (lucide), no shortcut assigned (all single-letter shortcuts are taken; not required by [`04_UI_GUIDELINES.md`](../../04_UI_GUIDELINES.md)).

Layout, top to bottom:

1. **Page header** (`components/page-header.tsx`, matching every other feature) — title "Project Init", description "Generate FDK phase scaffolds and AI project instruction files."
2. **Kind picker** — two large selectable cards (`tool-card.tsx`-style), "FDK Phase Scaffold" and "AI Project Instructions", each with a one-line description. Selecting one reveals its form below; the other card collapses to a compact "switch kind" affordance. No kind is pre-selected on first load (empty state, see §3).
3. **Generation form** (kind-dependent, see [05_COMPONENTS.md](05_COMPONENTS.md)):
   - *FDK Phase Scaffold*: phase number, phase name, objective. A live-updating "will generate: `Phase-09-Knowledge-Hub/`" folder-name preview.
   - *AI Project Instructions*: project name, description, tech stack (tag input), conventions (textarea, optional), output-file checklist (`CLAUDE.md` / `AGENTS.md` / `instructions.md`, at least one required).
   - A "Generate & Download" primary button, disabled until the form is valid. On submit: request fires, button shows a spinner state, and on success the browser downloads the zip immediately (no intermediate confirmation screen — generation has no destructive consequence, so no confirmation step is needed here, per [`04_UI_GUIDELINES.md §2`](../../04_UI_GUIDELINES.md) which reserves confirmation for destructive actions).
   - A read-only "Preview files" disclosure (collapsed by default) lists the filenames that will be generated for the current form state, each expandable to show its rendered content — lets a user sanity-check before downloading, without being an editor (per [01_SPEC.md §5](01_SPEC.md)).
4. **History section** — "Recent generations" list, newest first, max 20 items. Each row: kind icon, name, relative timestamp, "Download" and "Delete" actions. Delete requires the standard confirmation step (destructive action, per [`04_UI_GUIDELINES.md §2`](../../04_UI_GUIDELINES.md)).

## 2. Navigation

Added to `frontend/lib/nav-registry.ts` as a new `NavItem` (title "Project Init", href `/project-init`, icon `Sparkles`, description "Generate FDK scaffolds & AI instructions") — this is the shared nav registry, not Workbench's own code, so it does not touch the frozen Phase 01 surface. Automatically appears in the sidebar and the `cmdk` command palette, which both already read from this registry.

## 3. States

**Kind picker / no kind selected (initial empty state):** both cards shown, neither selected, no form, no "Generate" button visible yet. This is not an error — it is the expected first-load state, with a one-line hint ("Pick a template to get started").

**Form, unfilled:** "Generate & Download" disabled; inline validation messages appear per-field on blur, not as a blocking alert dialog (per [`04_UI_GUIDELINES.md §2`](../../04_UI_GUIDELINES.md)).

**Generating (loading):** button shows a spinner and "Generating…" label; form fields disabled for the duration (generation is synchronous and sub-second — no separate progress bar needed, unlike Ingest's async jobs).

**Generation error:** inline `Alert` (destructive variant) above the form with the server's error message and a "Try again" affordance; form fields remain filled (never cleared on error).

**History, empty:** `components/empty-state.tsx` (existing shared component) — "No generations yet" with a hint to use the form above; no separate call-to-action button needed since the form is already on-screen.

**History, loading:** skeleton rows (`components/ui/skeleton.tsx`), matching the pattern used elsewhere for TanStack Query `isLoading`.

**History, error:** inline retry affordance, consistent with other list views in the app.

**History, populated:** rows as described in §1.4.

## 4. Responsive behavior

Kind-picker cards stack vertically below `sm` breakpoint. Form fields are single-column at all widths (no dense table layout to adapt — this feature has no data-dense grid). History list rows collapse the timestamp to a compact form and keep both actions visible as icon buttons on narrow viewports, consistent with other list views in the app (e.g. Notes, Ingest job list).

## 5. Accessibility

- Kind-picker cards are real buttons (native `<button>` or `role="radio"` within a `radiogroup`), reachable and selectable via keyboard, not click-only `div`s.
- Form fields use plain `useState` with associated `<Label>`s, matching the pattern every existing Forge tool form already uses (e.g. `features/generators/password-generator.tsx`) — `react-hook-form`/`zod` are listed in the tech stack but not actually used by any shipped feature form; introducing them here would mean building a new shadcn `Form` primitive nobody else uses, which is unjustified for three simple fields. Correction to this doc's earlier draft, made once implementation revealed the assumption (§07_CODING_STANDARDS.md's documentation-in-code principle: match existing style, don't introduce a parallel pattern).
- The "Preview files" disclosure uses a native `<details>`-equivalent (shadcn `Accordion`) for keyboard operability.
- Delete confirmation dialog traps focus and is dismissible via `Escape`, matching the existing `components/ui/dialog.tsx` behavior used elsewhere.
