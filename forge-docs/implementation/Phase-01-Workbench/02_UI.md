## 1. Screens / views

This phase introduces one screen, in two modes:

- **Workbench home** (`/`) — **view mode.** The default state: a responsive grid of the visible, registered panels in their saved order. This is the sole home experience — "Dashboard" no longer exists as a separate concept (per [ADR-0001](../../decisions/0001-workbench-replaces-dashboard.md)).
- **Workbench home — customize mode.** Entered via a "Customize" toggle in the page header. Adds: per-panel show/hide controls, drag handles for reordering panels, and a "Manage pinned tools" entry point. Exiting customize mode returns to view mode; changes save immediately per action (no separate "Save" step), mirroring how Notes board edits already persist immediately.
- **Pin picker** — a dialog (not a new route) opened from customize mode or from the Pinned Tools panel itself, listing every entry from `frontend/lib/nav-registry.ts` plus the forward-looking Prompt Studio / Universal Converter entries (see §3.4), each with a pin/unpin toggle.
- **Search** (`/search`) — a new dedicated page, added per [ADR-0007](../../decisions/0007-search-dedicated-page.md), rendering the same result set the command palette already shows (secrets/notes/documents matches via `GET /api/search`), reachable as a pinned tool and as a standalone route.

## 2. Navigation

- The sidebar entry point that pointed at Dashboard now reads **"Workbench"**, same position, same icon (`LayoutDashboard`), same `D` shortcut, still targeting `/` — per [ADR-0001](../../decisions/0001-workbench-replaces-dashboard.md), this is a settled rename, not an open question.
- Reachable from the command palette (⌘K) exactly as before; "Customize Workbench" and "Manage pinned tools" are added as command-palette actions so the customize flow doesn't require finding the on-page toggle.
- Pinned tools in the Pinned Tools panel link directly to their target feature route, same as sidebar links — except disabled "coming soon" pins (Prompt Studio, Universal Converter), which are inert (see §3.4).
- The new `/search` page gets its own sidebar-adjacent reachability only via the Pinned Tools panel and the command palette for this phase — it is not added as a permanent top-level sidebar item (out of scope; see [`01_SPEC.md`](01_SPEC.md) §5).

## 3. States

### 3.1 Workbench home (view mode)

- **Loading:** skeleton cards in the last-known panel grid shape (or the default shape on first load) while `GET /api/workbench` resolves — matches the TanStack Query `isLoading` convention used elsewhere.
- **Error:** if the layout fetch fails, show an inline retryable error card in place of the grid — never a blank page. Individual panel data failing independently must not break the rest of the grid; each panel owns its own error state via the shared error boundary (per [`12_PANEL_INTERFACE.md`](12_PANEL_INTERFACE.md) §3).
- **Empty (no panels visible):** if every panel has been hidden, show a centered empty state with the copy "Every panel is hidden" and a primary action button that opens customize mode — never a truly blank page.
- **Populated:** the normal case — the panel grid in saved order.

### 3.2 Per-panel states

Every panel independently defines its own empty/loading/error/populated states (per [`12_PANEL_INTERFACE.md`](12_PANEL_INTERFACE.md) §1 — this is the panel's own responsibility, not something Workbench dictates centrally). Phase 01's five active panels:

| Panel | Empty state | Loading | Error |
|---|---|---|---|
| Pinned Tools | "No pinned tools yet" + button opening the pin picker | Skeleton row of icon placeholders | Inline retry, rest of grid unaffected |
| Recent Activity | "No activity yet" | Skeleton list rows | Inline retry |
| Quick Actions | N/A — never empty; fixed action set | Buttons render immediately (static, no network call) | N/A |
| Storage & System | N/A — always has a value once loaded | Skeleton stat blocks | Inline retry |
| Recent Notes | "No notes yet" + button linking to Notes | Skeleton cards | Inline retry |

Recent Projects (defined, not yet registered per [ADR-0005](../../decisions/0005-projects-primary-organizational-unit.md)) has no state to define in this document — it does not render at all until Phase 06 registers it, at which point Phase 06's own `02_UI.md` addendum (or this document's next revision) defines its states.

### 3.3 Customize mode

- **Entering:** panels gain drag handles and a per-panel visibility toggle; layout does not shift except for the addition of these controls.
- **Reordering:** drag-and-drop via `@dnd-kit` (same library as the Notes board), with an explicit keyboard-accessible fallback (arrow-key reorder while a panel is focused) — required per [`../../04_UI_GUIDELINES.md`](../../04_UI_GUIDELINES.md) §2 and the accessibility acceptance criteria in [`08_ACCEPTANCE.md`](08_ACCEPTANCE.md).
- **Hiding a panel:** toggling visibility off animates the panel out of the grid immediately and persists; it remains listed (dimmed) inside customize mode so it can be toggled back on.
- **Reset:** a "Reset to default layout" button, requiring one confirmation step (per [`../../04_UI_GUIDELINES.md`](../../04_UI_GUIDELINES.md) §2).

### 3.4 Pin picker

- **Loading:** N/A for the shipped-tool catalog (static); only the current pinned-set fetch can be loading, shown as skeleton checkmarks.
- **Empty:** never empty.
- **Disabled entries:** Prompt Studio and Universal Converter appear in the picker as pinnable (their pins can still be added/removed from the Pinned Tools panel per the default set in [`01_SPEC.md`](01_SPEC.md) §3, requirement 9) but are visually marked "Coming soon" and, once pinned, render as inert tiles in the Pinned Tools panel (no click target) rather than a dead link to a route that 404s.
- **Error:** if saving a pin/unpin fails, the toggle reverts optimistically and a toast reports the failure.

### 3.5 Search page

- **Empty query:** prompts for input, matching the command palette's empty state.
- **Loading:** skeleton result rows while `GET /api/search` resolves.
- **No results:** explicit "No matches for '…'" message, not a blank list.
- **Error:** inline retryable error, consistent with every other list-based feature.

## 4. Responsive behavior

- **Desktop:** multi-column panel grid (exact column count/breakpoints are a `05_COMPONENTS.md` implementation detail).
- **Mobile:** single-column stacked list of panels — reordering remains available via the same drag-and-drop + keyboard fallback described in §3.3.
- **Pin picker** is a full-screen dialog on mobile widths rather than a centered modal.
- **Search page** uses the same responsive list patterns as Vault/Notes' existing data-dense views.

## 5. Accessibility

- Every interactive element (panel visibility toggle, drag handle, pin/unpin toggle, quick-action buttons, reset button, search input/results) is reachable and operable by keyboard alone.
- Drag-and-drop reordering has a non-pointer path (arrow-key reorder) — a hard requirement, not a nice-to-have, carried into the acceptance criteria in [`08_ACCEPTANCE.md`](08_ACCEPTANCE.md).
- Panel visibility state and "coming soon" pin state are never communicated by color alone — both dimming and an explicit text label ("Hidden," "Coming soon").
- Pin picker toggles announce their state to assistive tech via native checkbox/switch semantics, not a custom div.
- Focus is explicitly managed when entering/exiting customize mode and opening/closing the pin picker (focus moves to the first meaningful control, and returns to the triggering element on close) — required per the accessibility acceptance criteria in [`08_ACCEPTANCE.md`](08_ACCEPTANCE.md).
