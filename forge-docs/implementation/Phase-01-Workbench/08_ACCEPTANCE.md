## 4. Performance acceptance criteria

- [x] Initial Workbench render completes in **under 1 second** on a local Docker Compose deployment (cold load through fully painted panel grid). — **Verified T16** via the Navigation/Resource Timing APIs against the (slower) dev server: navigation completes in ~462ms, `GET /api/workbench` resolves by ~644ms after navigation start. Measured against `next dev`, not the Docker Compose production build specifically (that build was confirmed to succeed, per §3, but not independently re-timed) — production is expected to be faster, not slower, so this comfortably clears the 1s bar either way.
- [ ] Drag-reorder interaction sustains **60 FPS** with no visible jank (measured per [`07_TESTING.md`](07_TESTING.md) §3). — **Not independently measurable this pass**: this automated browser session's compositor does not service animation frames at all (confirmed directly — `element.getAnimations()[0].currentTime` stayed at exactly `0` across 3+ real elapsed seconds, and a scheduled `requestAnimationFrame` callback never fired even after a full second), so no FPS reading taken here would be meaningful. This is a property of the automated test tool in this session, not of the app — dnd-kit's reordering uses CSS-transform-based drag, not layout-triggering changes, which is the correct approach for 60fps, but a real-device/real-browser measurement is needed to confirm empirically. **Tracked as [`QA/QA-0001-drag-performance.md`](QA/QA-0001-drag-performance.md)** — does not block Phase 01 sign-off, per the project owner's direction.
- [x] `PUT /api/workbench/layout` round-trips in **under 200 ms** on a local deployment. — **Verified T16** via Resource Timing: 18.3ms for a real toggle-triggered `PUT`, an order of magnitude under budget.
- [ ] No unnecessary re-renders occur during a drag interaction — the React DevTools Profiler shows only the dragged panel and its immediate neighbors re-rendering, not the full grid, not unrelated panels' own data-fetching hooks re-firing. — **Not verified at any point in this phase**: requires the React DevTools Profiler extension, which no automated browser session used across T9–T16 has had available. **Tracked as [`QA/QA-0001-drag-performance.md`](QA/QA-0001-drag-performance.md)** alongside the FPS criterion — does not block Phase 01 sign-off.

## 5. Accessibility acceptance criteria

- [x] Full keyboard navigation reaches and operates every interactive element in view mode, customize mode, and the pin picker. — **Verified T16**: real Tab-key walkthrough reached all 10 sidebar links in order, banner controls, and the Customize toggle; every panel's controls (Reorder handles, Hide/Show switches, Manage pinned tools, Reset to default) are genuine `<button>`/`role="switch"` elements reachable in tab order. Enter/Space *activation* of a focused native `<button>` is a browser platform guarantee for real keyboard input and doesn't depend on app code — this session's synthetic `Enter` keypress tool action produced a malformed event (empty `key` field, confirmed by instrumenting the button directly) and couldn't be used to prove activation the way the panel-reorder dnd-kit test could; this is the same class of testing-tool limitation T12 documented for its keyboard-sensor testing, not a gap in the app.
- [x] Keyboard-only drag-and-drop reordering works (arrow-key reorder while a panel is focused). — **Verified T16**: see §2 above — real, working end-to-end for both panel reorder and pin reorder.
- [ ] Focus management is correct: entering/exiting customize mode and opening/closing the pin picker moves focus to a sensible element and returns it to the trigger on close. — **Code-reviewed as correct, not live-verified this pass**: `app/(app)/page.tsx`'s `toggleMode()` schedules the focus shift inside `requestAnimationFrame`, which this automated browser session's tab never services at all (confirmed directly: a scheduled rAF callback never fired even after a full elapsed second — the same root cause as the frozen CSS-animation-clock finding below). The code follows the correct, standard pattern and is unchanged from T9's original implementation; a real browser tab (which services rAF every frame) would run it normally. **Tracked as [`QA/QA-0002-screen-reader-audit.md`](QA/QA-0002-screen-reader-audit.md)**'s keyboard-only pass — does not block Phase 01 sign-off.
- [x] Every icon-only or ambiguous control has a correct ARIA label (visibility toggle, drag handle, reset button, pin/unpin toggle). — **Verified T16** for every Workbench-owned control (`"Hide/Show <Panel> panel"`, `"Reorder <Panel>"`, `"Pin/Unpin <Tool>"`) via the accessibility tree. One unrelated, pre-existing icon-only control (the global mobile-nav hamburger trigger in `components/app-shell/`, hidden at desktop widths) was found to have **no** accessible name at all — flagged as a separate out-of-scope task (this predates Workbench and touches no file any Workbench task owns), not fixed here.
- [ ] Screen-reader pass (VoiceOver or NVDA spot check, or an equivalent automated tool) confirms panel titles, states ("Hidden," "Coming soon," loading/error states), and controls are announced correctly. — **Not performed this pass**: no VoiceOver/NVDA available in this environment. Partially substituted by the automated axe-core scan below (which does check accessible-name/role/state correctness, a meaningful chunk of what a screen-reader pass would catch) plus direct accessibility-tree inspection of every state (`"Hidden"`, `"Coming soon"` badges, empty/loading text all present with correct roles) — but a real assistive-technology pass is still outstanding. **Tracked as [`QA/QA-0002-screen-reader-audit.md`](QA/QA-0002-screen-reader-audit.md)** — does not block Phase 01 sign-off, per the project owner's direction.
- [x] An automated accessibility scan (axe or equivalent) against Workbench (both modes) and `/search` reports no critical/serious violations. — **Verified T16**: injected axe-core 4.12.1 directly into the live page and ran it scoped to Workbench's own `<main>` region — **zero violations** in view mode, customize mode, the pin picker dialog, and `/search`. A whole-page (unscoped) run surfaced two violations, but both are in pre-existing global app-shell chrome outside any file a Workbench task owns (sidebar footer text contrast in `components/app-shell/sidebar.tsx`; the command palette's `sr-only` dialog header not contained in a landmark, in `components/command-palette/`) — flagged as a separate out-of-scope task, not fixed here.

## 11. Genuine bug found and fixed during this pass

- **`AlertDialogAction` never closed its dialog.** `frontend/components/ui/alert-dialog.tsx`'s `AlertDialogAction` was a bare `Button` with no closing behavior — Base UI's `@base-ui/react/alert-dialog` has no auto-closing "Action" primitive (only `Close`), unlike Radix's `AlertDialog.Action`. This meant clicking "Reset" in the Workbench reset-confirmation dialog ran the reset mutation but left the dialog's `open` state stuck `true` forever, leaving an invisible, click-blocking, full-viewport overlay over the entire app. The same bug independently affected the pre-existing "Delete secret" confirmation in `frontend/features/secrets/secret-detail-sheet.tsx` — this was not a Workbench-introduced regression, it was a latent bug in a shared primitive that this phase's manual verification happened to be the first to exercise thoroughly. **Fixed** by rebuilding `AlertDialogAction` on `AlertDialogPrimitive.Close` (mirroring how `AlertDialogCancel` was already correctly implemented), so it now closes on click while still firing the caller's `onClick`. Verified: the dialog's `data-open`/`data-closed` state now correctly flips on click (confirmed via direct DOM inspection), where before the fix `data-open` never changed. `tsc`/`eslint`/`next build`/`docker compose build frontend` all clean afterward.

## 12. Environment limitations encountered during T16 verification (not app defects)

Documented here because they affected how several criteria above could be checked, and to avoid a future session re-discovering and re-investigating the same root cause:

- **This automated browser session's tab does not service animation frames.** `element.getAnimations()[0].currentTime` stayed at exactly `0` across 3+ real elapsed seconds (and after explicit `.finish()` calls), and a `requestAnimationFrame` callback scheduled directly never fired even after a full second. This is the single root cause behind three separate observations during this pass: dismissed dialogs appearing to "stick" visually after their state correctly flips to closed (CSS exit-animation-driven unmount never completes); the Workbench customize-mode focus-management code (which schedules its focus shift inside `requestAnimationFrame`) not visibly taking effect in this session despite being correct, unchanged code; and the general inability to measure drag-reorder FPS. None of this reflects a real browser tab, which services rAF/animation frames every ~16ms.
- **This session's synthetic `computer` key-press tool produces malformed keyboard events** (confirmed: dispatched `Enter`/`Space` presses arrive with an empty `key` field, so native `<button>` Enter/Space activation — a browser platform guarantee, not app behavior — doesn't fire). `dispatchEvent`-based synthetic events with a correctly-populated `key` worked for proving sensor wiring (e.g. dnd-kit's `defaultPrevented` check) but cannot trigger a browser's *default action* for untrusted events (also a web-platform rule, not fixable from either side). This is the same class of limitation T12's checkpoint documented for its own keyboard-sensor verification.
- **The `computer` tool's `screenshot`/`zoom` actions consistently timed out** in this session regardless of page state (tried on multiple pages, after reloads, with no pending dialogs) — verification relied on the accessibility tree (`read_page`), direct DOM/CSS inspection (`javascript_tool`), and network-timing APIs instead, all of which are reliable and don't depend on visual rendering.
