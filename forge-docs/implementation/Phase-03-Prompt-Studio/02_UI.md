> **Ownership:** Lead Software Engineer (phase-assigned).

## 1. Screens / views

**1.1 `/prompt-studio` — single screen, master-detail layout** (desktop ≥1024px: two panes side by side; see §4 for narrower widths):

- **Left pane — Prompt list**: search input (name/description), tag filter (multi-select chips, derived from tags already in use), "New Prompt" button, and a scrollable list of prompt cards (name, description excerpt, tags, current version number, last-updated timestamp).
- **Right pane — Prompt editor**, shown once a prompt is selected or "New Prompt" is clicked. Contains, top to bottom:
  - Metadata row: name (inline-editable text field), description (inline-editable), tags (tag input), version badge (e.g. "v4"), and a toolbar with **Duplicate**, **History**, and **Delete** actions.
  - **Variables panel**: a list of declared variables, each row showing name/type/required/default/description with edit-in-place controls, plus an "Add variable" row. Reorderable is not required (order in this list has no semantic meaning; the body references variables by name, not position).
  - **Body editor**: a `@monaco-editor/react` instance (plain text mode, no language server), showing the raw `${name}`-templated body. Declared variable names referenced in the body are highlighted distinctly from undeclared/unknown `${...}` tokens (the latter is exactly the condition that produces the save-time 422 in [`01_SPEC.md`](01_SPEC.md) §3.4 — the editor surfaces it before the user even attempts to save).
  - **Preview panel** (collapsible, open by default): one input per declared variable (pre-filled with its default, if any), and a read-only rendered-output box below that updates live as inputs change (client-side only, per [`01_SPEC.md`](01_SPEC.md) §3.5). A **Copy** button copies the rendered output. A required variable left blank shows its `${name}` token highlighted as unresolved in the output rather than rendering as empty text. Per the rendering-determinism constraint in [`01_SPEC.md`](01_SPEC.md) §3.16, this panel only ever performs substitution/escaping/validation — there is no way for a prompt body to express logic that changes what the preview does, so the preview's behavior is fully predictable from the body and the entered values alone.
  - **Save** / **Discard changes** appear only when the editor has unsaved edits to metadata, variables, or body (dirty-state tracking, matching the inline-edit pattern already used elsewhere in Forge rather than a separate "edit mode" toggle).

**1.2 Version History** — a slide-over panel (not a separate route), opened via the **History** toolbar action:
  - A list of versions (version number, timestamp, auto-generated note), newest first.
  - Selecting one version shows its full body/variables read-only, with a **Restore** button (confirmation required per §5/[`04_UI_GUIDELINES.md`](../../04_UI_GUIDELINES.md) §2).
  - Selecting two versions (checkbox-select, max 2) switches the panel to a diff view of their bodies, rendered with the existing `diff`-package-backed diff component already used by the Converters diff viewer.

## 2. Navigation

- New sidebar entry "Prompt Studio", plus a command-palette entry, both added via a single new `frontend/lib/nav-registry.ts` entry (existing file, existing array — no structural change to that file), per [`../../04_UI_GUIDELINES.md`](../../04_UI_GUIDELINES.md) §2. Suggested icon: `lucide-react`'s `Wand2` (distinct from Project Init's already-used `Sparkles`).
- Deep-linking a specific prompt (e.g. `/prompt-studio?id=<uuid>`) is supported so the command palette can, in a future enhancement, jump straight to a specific saved prompt — not required for this phase's initial command-palette entry, which only needs to open `/prompt-studio` itself.

## 3. States

**List (left pane):**
- **Empty** (no prompts exist yet): centered illustration/icon + "No prompts yet" + a primary "New Prompt" button — not blank space.
- **Loading**: skeleton rows (TanStack Query `isLoading`), matching the skeleton pattern used elsewhere (e.g. Project Init's history list).
- **Error**: inline retryable error banner in the list pane ("Couldn't load prompts — Retry"), not a full-page crash.
- **Populated**: prompt cards as described in §1.1; a zero-result state for an active search/tag filter ("No prompts match — Clear filters") is distinct from the true-empty state above.

**Editor (right pane):**
- **Empty** (no prompt selected, list is non-empty): a neutral placeholder ("Select a prompt, or create a new one").
- **Loading**: skeleton form (fetching a specific prompt's detail).
- **Error**: inline retryable banner if the detail fetch or a save fails; save errors surface as an inline message near the Save button plus a `sonner` toast, not a silent failure.
- **Populated**: the full editor as described in §1.1.

**Version History panel:**
- **Empty**: unreachable in practice (a prompt always has ≥1 version from creation), so no empty state is designed for "zero versions" — only the loading/error/populated states below.
- **Loading**: skeleton list.
- **Error**: inline retryable banner.
- **Populated**: version list / version detail / diff view as described in §1.2.

(Required per [`../../04_UI_GUIDELINES.md`](../../04_UI_GUIDELINES.md) §5.)

## 4. Responsive behavior

- **Desktop (≥1024px)**: side-by-side master-detail as described in §1.1.
- **Tablet/mobile (<1024px)**: single-pane, stack navigation — the list is the default view; selecting a prompt navigates to a full-width editor view with a back affordance (chevron/back button) to return to the list, rather than squeezing both panes into one narrow viewport. This matches the existing mobile-nav precedent noted in [`../../04_UI_GUIDELINES.md`](../../04_UI_GUIDELINES.md) §3 of "an explicit mobile layout, not just a squeezed desktop view."
- The Version History panel is a full-screen slide-over on mobile widths instead of a partial-width side panel.
- Minimum supported viewport: 375px (matches the existing mobile-nav baseline; no wider minimum is introduced by this phase).

## 5. Accessibility

- Every interactive element (search, tag filter, variable rows, Save/Discard, Duplicate/History/Delete, version selection, Restore, Copy) is reachable and operable via keyboard alone — no mouse-only affordance.
- The Monaco editor's own built-in keyboard accessibility is relied on as-is (no custom override); the variable-highlighting overlay described in §1.1 is decorative (color + underline) and must not interfere with screen-reader reading order of the underlying text.
- Confirmation dialogs (Delete, Restore) use the existing shadcn/Base UI dialog primitive, which already provides focus trapping and `Escape`-to-cancel — no new dialog implementation.
- The diff view pairs additions/removals with text markers (`+`/`-` prefixes or equivalent), not color alone, per [`../../04_UI_GUIDELINES.md`](../../04_UI_GUIDELINES.md) §4.
- An unresolved-required-variable indicator in the preview panel (§1.1) pairs a visual highlight with a text label (e.g. "missing"), not color alone.
- The `${name}` vs. literal-`$` escape hatch (`$$` per `string.Template`'s own rule — see [`README.md`](README.md) Risks) is documented as inline help text near the body editor, not only in this spec.
