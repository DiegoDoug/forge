Confirmed as a tooling/environment limitation, not an application bug, by cross-checking: plain `<input>` elements typed correctly via the identical tool throughout the session, and every affected Monaco/dialog interaction worked immediately once dispatched via a native DOM `.click()` or Monaco's own public model API. **Genuinely unverified, not just untried, and explicitly requested to stay on the T15/QA list by the project owner:** a human physically typing into Monaco via a real keyboard; Escape-to-close and focus-trap-on-open for `AlertDialog` confirmations (every dialog this session opened was via synthetic `.click()`, so the trusted-user-gesture path is unverified — though this is `AlertDialog`, a shared, unmodified component already used by Documents/Secrets/Project Init, so there's no evidence of a Prompt-Studio-specific defect); high-DPI scaling. File for a real human/device QA pass before or shortly after release.
