# Database

SQLite, accessed async via `aiosqlite` at runtime and synchronously via
plain `sqlite3` for Alembic migrations (see `alembic/env.py` — this split
avoids nesting an event loop inside uvicorn's already-running one when
migrations run from the FastAPI lifespan on startup).

## Schema

| Table | Purpose |
|---|---|
| `app_config` | Single row (`id=1`): master password hash, theme, setup timestamp |
| `folders` | Vault folders (self-referential `parent_id` for nesting) |
| `tags` | Vault tags (name + color) |
| `secrets` | Vault entries — `encrypted_value`/`encrypted_metadata` are PyNaCl ciphertext, never plaintext |
| `secret_versions` | Immutable snapshots written whenever a secret's value changes |
| `secret_tag_links` | Many-to-many join between `secrets` and `tags` |
| `notes` | Sticky notes — position/size/color/pin/archive + Markdown content |
| `notes_fts` | FTS5 virtual table over `notes.title`/`notes.content`, kept in sync by triggers |
| `documents` | Documents tab entries — title/pin + editor HTML content, stored as plain text (see [API.md](API.md) for export formats) |
| `documents_fts` | FTS5 virtual table over `documents.title`/`documents.content`, kept in sync by triggers |
| `activity_log` | Lightweight feed for the dashboard's "Recent activity" — never stores secret values or document contents |

Every table uses a `TEXT` primary key (a 32-char hex UUID from
`models/base.py::new_id`), not an auto-increment integer — this makes IDs
safe to expose in URLs/exports without leaking row counts, and makes backup
import (`services/settings/backup.py`) trivial (IDs round-trip exactly).

## Full-text search

`notes_fts` is an **external-content FTS5 table** (`content='notes',
content_rowid='rowid'`) — it stores no data of its own, just an index that
joins back to `notes` via SQLite's implicit `rowid`. Three triggers
(`notes_ai`/`notes_ad`/`notes_au`, created in the initial migration) keep it
in sync automatically; application code never has to remember to update the
index. `documents_fts` (added in `0002_documents.py`) follows the identical
pattern over `documents`.

Vault secrets are **not** in an FTS index — only `secrets.name` is
searchable (plain `LIKE`), and only the name; values stay encrypted and are
never indexed in plaintext.

## Migrations

Alembic-managed, in `backend/alembic/versions/`. The initial migration
(`0001_initial_schema.py`) builds every table from `SQLModel.metadata`
directly rather than hand-written `op.create_table()` calls — the models in
`app/models/` are the single source of truth for the starting schema, plus
the raw FTS5 DDL and triggers that SQLModel can't express.

**Every migration after that should be a normal, explicit Alembic revision**
(autogenerated via `alembic revision --autogenerate -m "..."` and reviewed,
or hand-written for anything SQLite's limited `ALTER TABLE` support can't
autogenerate — index/FTS/trigger changes, mostly). `render_as_batch=True` is
set in `alembic/env.py` so column-level changes use SQLite's
create-table-copy-drop dance under the hood automatically.

Migrations run automatically on backend startup
(`app/main.py::_run_migrations`) — there's no separate migration step in
deployment.

## Why SQLite, not Postgres

Forge targets self-hosting on modest hardware (a Raspberry Pi, a $5 VPS) by
a single user or small team. SQLite with FTS5 covers the search
requirements, WAL mode handles the concurrency Forge actually needs (a
handful of requests from one browser session), and it removes an entire
service (plus its backup/upgrade/connection-pool concerns) from the deploy.
If Forge ever needs multi-writer concurrency at real scale, that's a
deliberate future migration, not a default to hedge against now.
